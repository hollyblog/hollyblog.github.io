JVM 类加载和字节码优化：深入学习jvm类加载机制，掌握类加载器、字节码优化技术和性能调优方法，提升应用性能和稳定性。包括JIT编译优化、类加载与方法区调优。
## JIT编译优化
### JIT编译器概述
JIT(Just-In-Time)编译器是java虚拟机的核心组件之一，它在程序运行时将字节码编译为本地机器码，从而显著提升程序执行性能。

> JIT编译器通过运行时收集的性能数据，对热点代码进行激进的优化，这些优化在静态编译时是无法实现的。

- 性能提升：将解释执行的字节码编译为高效的本地机器码，性能提升10-100倍
- 动态优化：基于运行时数据进行优化，包括内联、逃逸分析和循环优化。
- 平衡策略：在编译时间和执行性能之间找到最佳平衡点。

### JIT编译原理
#### 编译触发机制
JIT编译器通过热点检测来识别需要编译的代码。当方法或者循环的执行次数达到特定阈值时，会触发编译过程。
- 方法调用计数器：记录方法被调用的次数
- 回边计数器：记录循环回边的执行次数
- 编译阈值：触发编译的计数器阈值
- 热度衰减：长时间未执行的代码热度会逐步降低。

#### 编译层次
HotSpot虚拟机采用分层编译策略：
- 层次0：解释执行（直接解释执行字节码，收集性能数据）
- 层次1-3：c1编译（客户端编译器，快速编译，基础优化）
- 层次4：c2编译（服务端编译器，深度优化，高性能代码）

### c1和c2编译器
#### c1编译器（客户端编译器）
c1编译器注重编译速度，能快速将字节码编译为本地机器码，适合启动性能要求较高的应用。
- 快速编译：编译速度快，启动时间短。
- 基础优化：进行基本的优化，如常量折叠、死代码消除
- 性能监控：收集详细的性能数据供c2编译器使用。
- 适用场景：桌面应用、短时间运行的程序。

#### c2编译器（服务端编译器）
c2编译器进行深度优化，生成高质量的本地代码，适合长时间运行的服务端应用。
- 深度优化：进行激进的优化，如内联、逃逸分析。对热点代码进行详细的分析和优化，包括循环展开、内联、逃逸分析等。
- 高质量代码：生成高性能的本地机器码
- 编译时间长：优化过程复杂，编译时间较长。
- 适用场景：服务器端应用、长时间运行的程序。
### 编译优化技术
#### 方法内联
方法内联是jit编译器最重要的优化技术之一，将方法调用替换为方法体的直接执行，消除了方法调用的开销。

是一种将调用方法的代码直接替换为被调用方法的代码的优化技术。它可以减少方法调用的开销，提高程序执行效率。

> 内联条件：方法体较小、调用频繁、非虚方法或可以去虚化的虚方法都是内联候选。

#### 逃逸分析
逃逸分析是一种分析对象是否会逃逸出其作用域的技术。在编译时，通过逃逸分析可以确定对象是否会被外部访问到。如果对象不会逃逸出其作用域，那么就可以对该对象进行优化，如栈上分配、标量替换等。

逃逸分析判断对象的作用域，如果对象不会逃逸出方法或线程，就可以进行栈上分配、标量替换等优化。

- 栈上分配：将对象分配在栈上而不是堆上。
- 标量替换：将对象拆分为基本类型变量
- 同步消除：消除不必要的同步操作。

#### 循环优化
循环是程序中的热点代码，jit编译器对循环进行多种优化。
- 循环展开：减少循环控制开销
- 循环不变量外提：将不变计算移出循环
- 循环向量化：利用simd指令并行执行
- 循环剥离：处理边界条件。

## 类加载与方法区调优

### 类加载机制深入
类加载是jvm将类的字节码文件加载到内存中，并转换为可执行代码的过程。
类加载器负责加载类的字节码文件，将其转换为jvm可以理解的类对象。类加载器按照一定的规则和顺序加载类，确保类的依赖关系被正确解析。
#### 类加载的生命周期
- 加载：通过类加载器加载类的字节码文件到内存中。
- 验证：验证类的字节码文件是否符合jvm规范，确保类的安全性。
- 准备：为类的静态变量分配内存，并将其初始化为默认值。
- 解析：将类中的符号引用转换为直接引用，确保类的方法调用和字段访问能够正确执行。
- 初始化：执行类的初始化代码，包括静态变量赋值和静态代码块的执行。
#### 双亲委派模型
双亲委派模型是类加载器的一种工作机制，它确保类的加载是安全的，避免类的重复加载。
双亲委派模型的工作原理如下：
- 当一个类加载器收到加载类的请求时，它首先将请求委托给父类加载器。
- 父类加载器继续向上委托，直到到达启动类加载器。
- 启动类加载器负责加载jvm核心类库，如java.lang包下的类。
- 如果启动类加载器无法加载类，它会将请求向下委托给子加载器。
- 子加载器负责加载自定义类加载器加载的类。
- 如果子加载器也无法加载类，它会抛出ClassNotFoundException异常。
双亲委派模型的优势在于：
- 避免类的重复加载：由于类加载器的委托机制，同一个类只会被加载一次。
- 确保类的安全性：类加载器的委托机制确保了类的加载是安全的，避免了恶意类的加载。
- 类加载器的加载顺序：启动类加载器（加载核心java类库）->扩展类加载器（加载扩展目录中的类库）->应用类加载器（加载应用类路径中的类库）->自定义类加载器。
### 方法区演进：从永久代到元空间
java8移除了永久代，引入了元空间，这一变化解决了永久代容易发生outofmemory error的问题，但也带来了新的调优挑战。

#### 永久代 vs 元空间
- 永久代：在java7及之前的版本中，存储类的元数据（如类名、方法信息、字段信息等）、常量池等；位于堆内存；大小固定，容易oom；gc效率低。
- 元空间：在java8及之后的版本中，只存储类的元数据；元空间是jvm的本地内存，与堆内存分离；大小可动态调整；减少oom风险。

#### 元空间优势
- 动态扩展：元空间大小可动态调整，避免了永久代固定大小的问题。
- 减少oom：由于使用本地内存，大大减少了oom的发生。
- 简化调优：不再需要精确设置永久代大小，调优更加简单。
- 提供性能：本地内存的使用提高了类元数据的访问性能。
- 与堆内存分离：元空间与堆内存分离，避免了堆内存满导致的元空间oom问题。

### 元空间调优实践
```java
-XX:MetaspaceSize=128m //元空间初始大小
-XX:MaxMetaspaceSize=256m //元空间最大大小(默认无限制)
-XX:CompressedClassSpaceSize=256m //压缩类空间大小(默认1gb)
-XX:+CMSClassUnloadingEnabled //启用类卸载(默认启用)
-XX:MetaspaceSize=128m //元空间gc触发阈值
```
> MetaspaceSize参数不是元空间的初始大小，而是首次触发元空间gc的阈值。设置合适的值可以避免频繁元空间gc。

```java
//使用jstat监控元空间
jstat -gc <pid> ls
//查看元空间详细信息
jcmd <pid> VM.metaspace
//使用jconsole或visualvm监控
Memory -> Non-Heap Memory -> Metaspace
```
#### 常见问题与解决方案
- 元空间oom：增加元空间大小，如-XX:MaxMetaspaceSize=256m；检查类加载泄漏；优化类加载策略。
- 频繁元空间gc：调整MetaspaceSize参数；减少动态类生成；启用类卸载。
- 内存使用过高：分析类加载情况；检查重复类加载；优化类路径配置。

### 类加载性能优化
启动性能优化策略
- 类预加载：在应用启动时，提前加载常用的类，减少运行时的类加载延迟。
- 类路径优化：精简类路径，移除不必要的jar包，减少类查找时间。
- 模块化设计：使用java9+的模块系统，实现按需加载。
- 延迟加载：非核心类实现延迟加载，减少启动时间。

类加载监控和分析
```java
//启用类加载日志
-XX:+TraceClassLoading //跟踪类加载过程
-XX:+TraceClassUnloading //跟踪类卸载过程
//详细的类加载信息

-XX:+UnlockDiagnosticVMOptions //解锁诊断选项
-XX:+TraceClassLoadingPreorder //打印类加载顺序
```
### 类加载问题排查
常见类加载问题
- ClassNotFoundException：类路径中找不到指定的类文件，检查类路径配置，确认jar包存在。
- NoClassDefFoundError：编译时存在但运行时找不到类。检查类是否被正确加载，是否存在依赖关系问题。
- LinkageError：类加载器冲突或者版本不兼容。检查类加载器层次，解决版本冲突。
- 类加载内存泄漏：类无法被卸载，导致内存泄漏；检查类加载器引用，启用“类卸载”。
